# Product Planning

## Overview

市場調査や大まかなアイデアを、そのまま作れるプロダクト案に圧縮する。作るべきもの、作らないもの、最初の勝ち筋を明確にしてから実装へ渡す。

## Workflow

1. 入力を整理する。
   - `trend-research` の結果、ユーザー条件、事業制約、開発期間、得意技術を確認する。
   - 入力が曖昧でも、意思決定に必要な観点を補って整理する。
2. 誰の何を解くかを定義する。
   - 最初の顧客像を一段深く具体化する。
   - 課題が頻繁か、痛みが強いか、支払い意欲につながるかを確認する。
3. プロダクト案を圧縮する。
   - 提案を「コア価値」「主要ユースケース」「差別化」「作らないもの」に分ける。
   - 似た案が複数あるなら、最も wedge が強いものに寄せる。
4. MVP を切る。
   - 最初に絶対必要な機能と、後回しにできる機能を分ける。
   - MVP は短期間で検証可能であることを優先する。
5. 実装へ渡す形にする。
   - `builder` と `gstack-plan-eng-review` がすぐ動ける粒度まで落とす。
   - 必要なら success metric、初期ターゲット、主要画面、主要フローを明記する。

## Output

- 最初に推奨するプロダクト案を1つに絞る。
- 次に以下を整理する。
  - 対象ユーザー
  - 解く課題
  - なぜ今か
  - 競合に対する差別化
  - MVP に含めるもの
  - MVP から外すもの
  - 最初の success metric
  - 実装へ渡す次アクション
- 必要なら 2 番手案も短く添える。

## Guardrails

- 調査結果をそのまま列挙して終わらせず、必ず1つの提案に収束させる。
- 機能を盛り込みすぎず、最初の検証に必要な最小範囲へ切る。
- 競争が激しい市場では、差別化が弱い案をそのまま採用しない。
