# Trend Research

## Overview

学習済み知識ではなく、今真実であることを調べる。どのアプリ案に作る価値があるかを判断するための最近の証拠を重視する。

## Workflow

1. Frame the research question.
   - 対象ユーザー、地域、カテゴリ、期間、調査が支えるべき意思決定を定義する。
   - 「良いアプリ案を見つけて」のような曖昧な依頼を、評価軸が明確な問いに変換する。
2. Gather current evidence.
   - 不安定な事実、需要シグナル、競合情報、価格、ランキング、ローンチ動向、現行 API などは web browsing を使う。
   - 一次情報、公式サイト、アプリストア、公開ローンチボード、高信号の分析記事を優先する。
   - 重要な主張には必ず日付とリンクを付ける。
3. Evaluate opportunities.
   - 各案を需要シグナル、緊急度、競争密度、実装難度、収益化経路、MVP までの速さで評価する。
   - 明確な勝ち筋がない過密市場は落とす。
   - 最近の変化が参入の楔になる案を優先する。
4. Turn research into action.
   - 調査結果の羅列ではなく、上位の機会を推薦する。
   - 各案が今 viable な理由と、まだ不足している証拠を示す。
5. Hand off cleanly.
   - 企画や実装スキルがそのまま使える形で出力する。

## Output

- 最初に意思決定の問いを書く。
- 次に機会案を順位付きで並べる。
- 各案には以下を含める。
  - ユーザーと課題
  - なぜ今か
  - 日付とリンク付きの証拠
  - 競合に対する勝ち筋
  - MVP の提案
  - 主なリスク
- 最後に、次に作るべき案を1つに絞って示す。

## Guardrails

- 最新性が重要なときは記憶で答えず、必ず browse する。
- 人気と需要の質を混同しない。支払い意欲や反復する痛みの証拠を見る。
- 生リンクだけを並べて終わらせず、必ず要約して判断につなげる。
