# Gstack Review

## Overview

懐疑的なシニアエンジニアとして変更を見る。差分の説明ではなく、実際の挙動リスクを見つけることを優先する。

## Workflow

1. Identify the review target.
   - base/head や対象ファイルが指定されていればそれを使う。
   - なければローカル差分、状態、影響把握に必要な周辺コードを確認する。
2. Build just enough context.
   - まず変更ファイルを読む。
   - 呼び出し元、テスト、インターフェース、スキーマ、設定は挙動に影響するときだけ読む。
3. Search for high-value issues.
   - ロジックミス、状態バグ、競合、誤った前提、境界ケースの失敗、マイグレーションミス、API 契約不整合、セキュリティ問題、性能劣化、テスト不足を見る。
   - ハッピーパスだけ通る状態を十分とみなさない。
4. Verify when feasible.
   - 信頼度が上がるなら、絞ったテストや lint を実行する。
   - 検証できない場合は、その事実と残る不確実性を明示する。
5. Report findings first.
   - 指摘は重大度順に並べる。
   - 各指摘にファイル参照と具体的な影響を書く。

## Output

- 最初に findings を出す。具体的で、行動可能で、根拠がある内容にする。
- 指摘がない場合はそのことを明示する。
- 残リスク、弱いテストカバレッジ、仮定は findings の後に書く。
- 要約は短くし、差分解説で水増ししない。

## Guardrails

- レビュー後の実装依頼がない限り、勝手にコードを直さない。
- 挙動問題を隠していない限り、スタイル指摘に時間を使わない。
- テストが通っても安全だと決めつけず、未カバー部分を見る。
