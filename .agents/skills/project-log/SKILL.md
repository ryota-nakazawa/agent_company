# Project Log

## Overview

プロジェクトの記憶を作る担当として振る舞う。後から「なぜそうしたか」「何を確認したか」「次に何をするか」がすぐ追える、短く高密度な Markdown ログを残す。

## 記録先

- ルートの [logs/README.md](/Users/ryota/Desktop/エージェント作成/agent_team/logs/README.md) に従う。
- 基本の保存先は `logs/YYYY-MM-DD-<topic>.md`。
- 同じ作業の追記なら新規作成より既存ログの更新を優先する。

## Workflow

1. 記録対象を決める。
   - どのタスク、どの変更、どの意思決定を記録するのかを短く定義する。
   - 1ログに複数テーマを混ぜすぎない。
2. 事実を集める。
   - 依頼内容、調査結果、決定、実装内容、検証結果、残課題を整理する。
   - 推測ではなく、実際に起きたことを書く。
3. ログを構造化する。
   - 少なくとも `概要`、`判断`、`実施内容`、`検証`、`次アクション` を含める。
   - 外部情報があるなら出典リンクと日付を入れる。
4. 更新単位を保つ。
   - だらだら追記せず、意味のある節目でまとめる。
   - 既存ログに追記した場合も、今回何を追加したか分かるようにする。

## Output

- 実際の Markdown ログファイルを作成または更新する。
- ユーザーへの返答では、保存先パスと今回記録した要点だけを短く伝える。

## Guardrails

- 感想文にしない。再実行や引き継ぎに必要な情報を優先する。
- ログが長くなりすぎるなら、新しい節目で別ファイルに分ける。
- 機密情報や不要な生ログをそのまま貼らない。
