# Ui Polish

## Overview

実装済みの機能を、より理解しやすく使いやすい画面に仕上げる。機能追加よりも、情報の見せ方、視線誘導、余白、配色、モバイル時の破綻防止を優先する。

## Workflow

1. 画面の役割を定義する。
   - この画面でユーザーに何を理解させ、何を行動させたいかを一文で言い換える。
   - 目的が曖昧なら、先にプロダクトのコア価値へ戻る。
2. 現状の問題を特定する。
   - 情報過多、視線誘導不足、CTA の弱さ、余白不足、配色の弱さ、モバイル崩れ、アクセシビリティ不足を見る。
   - 既存のデザインシステムがあれば尊重する。
3. 改修方針を決める。
   - まず情報階層、次にレイアウト、最後に装飾を整える。
   - 必要ならセクションの順番を組み替える。
4. 実装する。
   - タイポグラフィ、色、余白、カード、ボタン、状態表示、レスポンシブを調整する。
   - 機能面に影響しない範囲で視認性と理解速度を上げる。
5. 確認する。
   - デスクトップとモバイル幅の両方で破綻がないかを見る。
   - 実ブラウザ確認がない場合でも、CSS と DOM 構造から明らかな破綻がないか点検する。

## Output

- 何を改善したかを、見た目ではなく効果ベースで説明する。
- 必要なら次に `gstack-review` または `browser-qa` に回す前提を明示する。

## Guardrails

- 機能を勝手に増やしすぎない。
- 派手さだけを足さず、理解しやすさを優先する。
- 既存の一貫性を壊す大改造は、明確な理由があるときだけ行う。
